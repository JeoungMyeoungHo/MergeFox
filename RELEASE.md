# Release process

This document describes how to cut a new mergeFox release. Packaging is
driven by [`cargo-dist`](https://axodotdev.github.io/cargo-dist/) â€”
config lives at `dist-workspace.toml`, and the release workflow at
`.github/workflows/release.yml` is **autogenerated** (run
`dist generate` to refresh).

## What ships

Each release produces:

| Artifact | Platform | Format |
|---|---|---|
| `mergefox-aarch64-apple-darwin.tar.xz` | macOS Apple Silicon | compressed tarball |
| `mergefox-x86_64-apple-darwin.tar.xz` | macOS Intel | compressed tarball |
| `mergefox-x86_64-unknown-linux-gnu.tar.xz` | Linux x64 | compressed tarball |
| `mergefox-aarch64-unknown-linux-gnu.tar.xz` | Linux arm64 | compressed tarball |
| `mergefox-x86_64-pc-windows-msvc.zip` | Windows x64 | zip |
| `mergefox-x86_64-pc-windows-msvc.msi` | Windows x64 | MSI installer |
| `mergefox-installer.sh` | macOS + Linux | `curl \| sh` oneliner |
| `mergefox-installer.ps1` | Windows | `iwr \| iex` oneliner |
| `source.tar.gz` | â€” | git archive for audit |
| `sha256.sum` | â€” | checksums for everything above |

**DMG** is not produced in the alpha because unsigned DMGs provide no
UX upgrade over the tarball (Gatekeeper blocks both the same way) and
Apple Developer ID signing is out of scope until beta. When signing
lands, add `dmg` to `installers` in `dist-workspace.toml` and
re-run `dist init`.

## Dependencies users must install

mergeFox shells out to the system `git` binary for every write op. It
is **not** bundled. Platform installers note this in their README,
and the app shows a startup notification if `git` isn't on PATH.

| OS | How to install git |
|---|---|
| macOS | Xcode Command Line Tools (`xcode-select --install`) or `brew install git` |
| Windows | [Git for Windows MSI](https://git-scm.com/download/win) |
| Linux | `apt install git` / `dnf install git` / `pacman -S git` |

The `.deb` / `.rpm` variants (planned) list `git (>= 2.30)` as a
package dependency so it installs automatically.

## Cutting a release

```bash
# 1. Bump the version
$EDITOR Cargo.toml CHANGELOG.md           # Cargo.toml `version = ...` + CHANGELOG Unreleasedâ†’X.Y.Z

# 2. Commit
git commit -am "release: v0.1.0-alpha.2"

# 3. Tag + push (the tag trigger runs the release workflow)
git tag v0.1.0-alpha.2
git push origin main --tags
```

Tag format: `vMAJOR.MINOR.PATCH[-PRERELEASE]`. Anything with
`-alpha.N`, `-beta.N`, or `-rc.N` becomes a GitHub Pre-release.

The tag push starts three jobs in parallel (`plan`, `build-local`,
`build-global`) and ends with a single draft GitHub Release. Review
the draft, hit Publish.

## Local validation before tagging

```bash
dist plan                 # dry-run â€” prints what would be built
dist build                # actually build for the host platform
dist build --target aarch64-apple-darwin x86_64-apple-darwin  # multi
```

Outputs land in `target/distrib/`. Test at least the host-platform
tarball + installer script before pushing the tag.

## Changing the build matrix

Edit `dist-workspace.toml`:

```toml
targets = [
    "aarch64-apple-darwin",
    "x86_64-apple-darwin",
    "x86_64-pc-windows-msvc",
    "x86_64-unknown-linux-gnu",
    "aarch64-unknown-linux-gnu",
]
installers = ["shell", "powershell", "msi"]
```

Then regenerate CI:

```bash
dist init --yes
```

This rewrites `.github/workflows/release.yml` â€” commit the changes
alongside whatever config you edited.

## Code signing (planned for beta)

Signing hooks will be wired once we have certificates. The shape:

### macOS
- Apple Developer Program membership ($99/year)
- Developer ID Application certificate
- App-specific password for `notarytool`
- Secrets on GitHub:
  `APPLE_SIGNING_CERT`, `APPLE_SIGNING_CERT_PASSWORD`,
  `APPLE_SIGNING_IDENTITY`, `APPLE_NOTARIZE_USER`,
  `APPLE_NOTARIZE_PASSWORD`, `APPLE_TEAM_ID`.

### Windows
- OV or EV code signing certificate
- Secrets on GitHub:
  `WINDOWS_SIGNING_CERT`, `WINDOWS_SIGNING_CERT_PASSWORD`.

`cargo-dist` supports these via the `ssldotcom-windows-sign` and
`apple-codesign` integrations â€” see the cargo-dist book for the full
toml layout when we're ready to enable them.
